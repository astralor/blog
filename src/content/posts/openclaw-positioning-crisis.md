## 一、一周反转

2026 年 3 月的第二周，中国互联网发生了一件有趣的事。

闲鱼上，"OpenClaw 上门安装"的商品还没下架，"OpenClaw 上门卸载"的服务就已经挂了出来。安装报价 500 元，卸载只要 15 块。从抢着装到急着删，中间隔了不到一周。

OpenClaw 是一个开源 AI Agent 框架，因为图标是一只红色龙虾，被中文互联网昵称为"小龙虾"。它能接管你的电脑，通过聊天软件接收指令，自主完成文件管理、代码编写、邮件处理等任务。GitHub 上 25 万星标，超越了 React，成为平台历史上增速最快的开源项目[^1]。黄仁勋在摩根士丹利的大会上说，这可能是"历史上最重要的软件发布之一"[^2]。

但在中国市场，这个"历史上最重要的软件"，正在经历一种近乎荒诞的生命周期：花 500 块请人装上，用了三天，再花 15 块请人卸掉。

这不是一个"产品不好所以卸载"的故事。OpenClaw 的技术底子是扎实的，生态是活跃的，否则不会有 13 家大厂争相入局。问题出在别的地方——**它从来没有建立起准确的产品定位，而这个空白被外部力量以各自的方式填满了。**

## 二、三层定位，从未对齐

分析 OpenClaw 的困境，需要区分三个视角。

### 创造者视角：这是基础设施

Peter Steinberger 做 OpenClaw 的出发点很明确：给懂技术的人造一个本地优先的 AI Agent 框架。

从产品设计决策可以看出这个定位：安装方式是 `npm install`，假设用户懂命令行。默认绑定 `0.0.0.0:18789` 监听全网卡——这是开发便利性优先的选择，不是消费级产品的做法。权限模型是全有或全无，假设用户理解自己在授权什么。官方 Discord 里有维护者直接说过一句话，很能说明问题：

> "If you can't understand how to run a command line, this is far too dangerous of a project for you to use safely."[^3]

OpenClaw 的自我定位是 Linux 级别的基础设施。这个定位在开发者社区完全成立——25 万星标中的前 5 万，大概率都是能驾驭它的人。

### 市场视角：这应该是消费级产品

但从 1 月底开始，三股外部力量同时改写了这个定位。

**第一股是云厂商的平台战争。** 阿里云一键部署、腾讯 QClaw 一键启动包、字节 ArkClaw、百度移动版、小米 Xiaomi miclaw——13 家大厂在两个月内全部入局[^4]。它们传递的信号高度一致："人人都能用。"腾讯说"一键部署，微信遥控"，阿里说"每天限量 9.9 元"，百度说"新用户免费体验 10 天"。这些话术把一个开发者框架重新包装成了消费级服务。

大厂的动机不难理解——它们在抢 AI Agent 生态的流量入口。但它们解决了"装上去"的问题，没有解决"用得了"和"用得安全"的问题。

3 月 6 日，腾讯在深圳总部搞了一场免费安装活动，近千人带着电脑排队[^5]。马化腾在朋友圈转发相关新闻，感慨"没想到这么火"。这个场景很有画面感，也很说明问题——排队的人里，有多少人理解他们即将在电脑上安装的东西需要 root 级别的系统权限？

**第二股是"养虾" meme 的传播力。** 从产品定位角度看，"养虾"这个词做了一件极强又极危险的事——它把一个技术工具变成了一种社交行为。"养"暗示情感连接和长期陪伴，"虾"把技术感消解了。你不会说"我今天部署了一个分布式 Agent 框架"，但你会说"我今天养了只虾"。叙事从"你需不需要它"变成了"你有没有它"。

这种 meme 化把 OpenClaw 的品类归属从"AI 开发工具"挪到了"数字宠物 / 社交货币"——跟当年抢购 Switch 健身环是同一个心理机制。

**第三股是安装服务商的利益驱动。** 闲鱼 500 元上门安装、小红书 9.9 元课程、199 元付费社群——这些服务商本质上是 OpenClaw 的非官方分销渠道。它们的利润来自安装量而不是留存率，所以没有动机去做用户筛选。网传有人几天赚了 26 万[^6]，数字不一定真实，但这个叙事进一步放大了入场冲动。

三股力量叠加的结果是：**大量用户跳过了"我为什么需要它"和"我能不能驾驭它"这两步，直接进入了"我拥有它"。**

![三层定位从未对齐：开发者在上面安静写代码，市场在中间大喊"人人可用"，用户在下面仰望一个虚幻的 Jarvis](./images/openclaw-positioning-gap.png)

### 用户视角：我以为它是 Jarvis

当一个普通用户听到"AI 助手，能帮你发邮件、写代码、管文件、24 小时在线"，他脑子里浮现的参考系是钢铁侠的 Jarvis——无需说明书、开箱即用、永远理解你的意图、绝不犯错。

但实际到手的是一个纯白板程序，连自己是谁都不知道。差评 X.PIN 有篇文章描述得很准确："刚装好的 OpenClaw，只有一个名叫 Soul.md 的文件是完整的，限定了最底层的底线：温和、友善、别干坏事。除此以外，它什么都不知道。"[^7]

一位律师花了 7 个小时才跑通安装流程——一边用 Coze 写代码一边让 GPT 解 Bug[^8]。一个产品经理用了几周才把它调教到能用的状态，每月花费几百美元 token 费，自嘲"贷款上班"。更多人可能在第一天就遇到了 Node.js 版本冲突或 API key 配置错误，然后再也没有打开过。

用户期望的是成品，拿到的是需要自己完成最后 80% 工作的开发套件。这个落差才是真正致命的。

## 三、信任的崩塌

如果只是"不好用"，用户的反应是"算了，放着不管"，不至于恐慌卸载。

引爆卸载的是**信任链条的连续断裂**。

第一环是技术信任。2 月下旬，Meta 超级智能实验室的对齐与安全总监 Summer Yue 在 X 上发帖，说她让 OpenClaw 帮忙整理邮箱，特意叮嘱"只看不操作"。结果邮件太多触发了上下文压缩，模型把"只看不操作"的指令遗忘了，然后开始疯狂删除她的邮件。"I had to RUN to my Mac mini like I was defusing a bomb"，她写道[^9]。

这个故事在全球科技媒体上广泛传播。它的杀伤力在于——如果连 Meta 的 AI 安全总监都翻车，普通人怎么可能安全使用？

技术层面的问题远不止这一个。安全研究机构陆续公开了一系列发现：13.5 万个 OpenClaw 实例暴露在公网上，其中 5 万多个存在远程代码执行漏洞[^10]。ClawHub 技能市场上超过 820 个恶意插件，占整个市场的约 20%，主要用于部署 macOS 信息窃取器[^11]。3 月 8 日，JFrog 发现了一个名为 GhostClaw 的恶意 npm 包，伪装成 OpenClaw 安装器，能窃取系统密码、SSH 密钥、浏览器密码甚至加密货币钱包助记词[^12]。

第二环是权威信任。3 月 8 日，工信部网络安全威胁和漏洞信息共享平台发出安全预警[^13]。3 月 10 日，国家互联网应急中心（CNCERT）发布正式风险提示，新华网和央视同步转发[^14]。提示点名了四类风险：提示词注入、误操作、插件投毒、安全漏洞。

对中国用户来说，官方警告的效力是核弹级的。之前大厂背书带来的安全感——"腾讯都在推，肯定靠谱"——瞬间反转为"连国家都说不安全了"。

第三环是社交信任。"有人的邮件被 AI 删光了"这个故事在朋友圈和群聊中快速传播。注意，这些故事不需要是发生在传播者身上的，也不需要完全准确——它只需要**可信**就够了。因为大多数用户本来就不理解 OpenClaw 在后台做什么，所以任何负面叙事都无法被反驳。

三环断裂之后，用户面临的决策其实很简单：**装 OpenClaw 的收益是模糊的（"可能有用"），不卸载的风险是具体的（"可能泄露密码"）。** 当收益模糊而风险具体时，损失厌恶会驱动用户选择消除风险。

而且卸载的成本（15 块）远低于安装的成本（500 块），这个不对称进一步降低了决策摩擦。

## 四、一个品类的困境

如果只是分析 OpenClaw 自身的问题，这篇文章写到这里就够了。但我更在意的是这件事指向的东西——**AI Agent 作为一个产品品类，有一些结构性的定位困难，不会因为某一款产品做得更好就消失。**

### 权限悖论

Agent 有用是因为它有权限。它能帮你发邮件，是因为你授权了它访问你的邮箱。它能帮你管文件，是因为你给了它文件系统的读写权。能力和风险是同一枚硬币的两面。

传统软件的定位边界是清晰的：Photoshop 是修图的，Excel 是算表的。用户知道它们**不会做什么**。但 AI Agent 的价值主张是"什么都能做"——而"什么都能做"的推论是"什么都可能做错"。

你很难给一个"什么都能做"的产品写出简洁的定位。它不像"修图工具"或"电子表格"那样能被一句话概括。而一个无法被简洁定位的产品，就注定会被不同的人以不同的方式理解——也误解。

### 信任的冷启动

工具的心智模型是：拿起它 → 用它 → 放下它。用户全程有控制感。

Agent 的心智模型更接近：雇一个人 → 给它任务 → 它自己决定怎么做 → 你验收结果。这需要信任，而信任需要时间。

问题是，OpenClaw 需要在第一天就获得 root 级别的权限才能工作。没有"先让它做点小事、确认没问题、再给更多权限"的渐进路径。这就像面试当天就要求新员工拿到公司所有系统的 admin 权限。

2026.3.2 版本试图修正这一点，把默认的 tools.profile 收紧为 messaging，屏蔽了文件操作、命令执行等高风险功能[^15]。方向是对的，但来得太晚，而且方式粗暴——老用户升级后所有功能突然失效，掘金上"升级后傻得一批"的帖子被广泛传播。

### Token 的隐性成本

另一个定位层面的问题是成本。

"开源"和"免费"是两个不同的词，但大多数非技术用户会把它们画等号。OpenClaw 是开源免费的，但运行它需要持续付费购买大模型 API token。有人两周花了 254 美元，有人一个月 800 美元——这些都不是极端使用场景下的数字[^16]。默认配置下，光是心跳机制（每半小时让 Agent 检查一次状态）就能消耗大量 token[^17]。

如果产品在定位层面就明确传达"这是一个每月需要投入几十到几百美元的服务"，用户的预期会完全不同。但"开源免费"的叙事把这个信息遮蔽了，用户发现真相时感受到的不是"价格合理"，而是"被骗了"。

### 开源产品的定位权悖论

最后一个结构性问题：谁来定义这个产品是什么？

传统产品的定位由产品方控制。苹果决定 iPhone 是什么，通过营销、体验设计和封闭生态来统一用户认知。但 OpenClaw 是开源的，它的定位被分散到了无数个第三方手里：大厂说它是"AI 助手"，闲鱼说它是"值得花 500 块装的东西"，小红书说它是"不装就落伍的社交货币"，工信部说它是"安全隐患"。

每一个第三方都在用自己的利益框架重新定义 OpenClaw。而 OpenClaw 官方既没有能力也没有强烈动力去统一这些叙事——它毕竟是一个开源项目，不是一家要对用户负责的公司。

**这可能是开源 AI Agent 面临的最独特的困境：如果你不定义自己，别人会替你定义。而替你定义的人，不一定跟你的用户利益对齐。**

### 所有问题的共同根因

权限悖论、信任冷启动、成本黑洞、定位权丧失——这些问题看起来各自独立，但往下挖一层，根因是同一个：**当前的大模型能力还不足以支撑 AI Agent 的终极承诺。**

AI Agent 的终极目标是明确的：全托管。用户描述意图，Agent 自主规划、执行、纠错、交付结果。这个愿景没有任何问题，它就是 AGI 路线图上的必经之路。

但现实是，今天的模型做不到这一点。上下文窗口会溢出导致指令遗忘（Summer Yue 的邮件就是这么被删的），多步推理会累积误差，工具调用会产生幻觉，长时间自主运行会偏离初始意图。这些不是 OpenClaw 的工程 bug——同样的底层模型换一个框架，同样的问题依然存在。

工程层面的问题（权限管理、错误恢复、成本控制、安全沙箱）是可以持续迭代的，事实上 OpenClaw 也在快速迭代。但模型能力是瓶颈。当模型还无法可靠地完成全托管时，产品就必须基于这个现实来设计，而不是把终局愿景当作今天的卖点。

**OpenClaw 的割裂恰恰在这里：它用终局的愿景做了今天的定位。** 用户被"AI 替你干活"的承诺吸引来，却发现自己需要花大量时间教它干活、盯着它干活、以及收拾它干错的活。这不是 OpenClaw 做得不好，而是整个品类在模型能力到达临界点之前，必然面对的结构性落差。

## 五、三个明确的判断

前面分析了问题，这一节我想给出自己的观点。

### 全托管是终局，但产品必须基于当下设计

AI Agent 的方向是对的。模型能力在进步，上下文管理在改善，工具调用的可靠性在提升。我相信全托管的 Agent 终将到来——也许不是明年，但它在 AGI 的路线图上。

但"方向正确"不等于"现在就能按终局来卖"。

一个诚实的 AI 产品，应该基于模型**当前**的能力边界来设计交互、设定预期、规划权限。而不是拿一个还需要三年才能成熟的愿景去做今天的用户承诺。OpenClaw 对技术用户是一个优秀的框架，但对大众用户，它推销了一个今天兑现不了的未来。

这不是悲观——恰恰相反，正因为相信 Agent 一定会成熟，才更需要在早期建立正确的用户预期。过度承诺带来的信任崩塌，比缓慢建立信任的代价要大得多。"养虾"潮的反转已经证明了这一点。

### 权限分层和安全智能体不是可选项

对于绝大部分用户来说，两件事是刚需：

**第一，权限必须分层。** 用户不应该从 root 权限开始使用 Agent。正确的路径是：只读模式（Agent 能看、能分析、能建议）→ 确认执行模式（Agent 提出操作方案，用户逐条批准）→ 自主执行模式（Agent 在授权范围内自主行动）。每一层都让用户感受到价值的同时建立信任。这不是可选的安全特性，而是产品能不能活过第一周的基础设施。

**第二，需要一个"安全智能体"。** 一个 24 小时自主运行的 Agent 旁边，应该有另一个 Agent 在看着它——类似生产环境中的 watchdog。当主 Agent 的行为偏离预设边界（大量删除文件、异常 API 调用、token 消耗飙升），安全智能体应该能自动拦截并通知用户。

目前 OpenClaw 的架构里，这两个东西都是缺失的。用户自己盯着 Agent 干活，本质上就违反了 Agent 的价值主张——你雇了一个员工，结果你得全程站在他背后看着他工作。

### 做产品，就必须画清边界

开源项目可以"自由使用，后果自负"——这是开源精神的一部分，无可厚非。OpenClaw 作为开源框架不一定需要把"不适合谁"说得很清楚。

但**一旦有人基于 OpenClaw 做产品**——无论是大厂的一键部署服务，还是创业公司的 Agent SaaS——就必须把边界画清楚。"这个产品适合谁、不适合谁、能做什么、不能做什么、会花多少钱、有什么风险"，每一条都要在用户决策之前明确传达。

尤其是在企业场景下。个人用户的 Agent 犯错，最坏是丢几封邮件。但企业环境里，Agent 犯错的代价是商业数据泄露、合规事故、客户信任崩塌、甚至法律责任。在企业 AI Agent 产品的说明书里，**"不适合谁"可能比"适合谁"更重要。**

这不是胆小，是负责。

## 结语

从"养虾"到"卸虾"的一周，不是一个失败的故事。它是 AI Agent 这个品类在走向成熟的路上必须经历的一次碰撞——用户的真实预期，第一次跟产品的真实能力发生了大规模的正面接触。碰撞会疼，但碰撞之后大家会更清楚彼此的位置。

模型在进步，工程在完善，生态在成熟。全托管的 AI Agent 会来的。但在它到来之前，做产品的人有责任基于今天的现实——而不是明天的愿景——去设计产品、设定预期、建立信任。

OpenClaw 的故事告诉我们：**方向对了不够，节奏也得对。**
