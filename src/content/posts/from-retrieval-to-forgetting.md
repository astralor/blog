> 我给 AI 配了一套完整的记忆搜索系统——向量索引、混合检索、语义搜索——结果 87% 的对话里，它根本不用。

## 一个反直觉的发现

最近在分析 OpenClaw 的运行数据时，我发现了一个有趣的现象。

OpenClaw 是一个开源 AI Agent 框架，内置了相当完善的记忆系统：Voyage embedding、BM25 + 向量混合检索、SQLite 索引、时间衰减……技术栈拉满。系统提示里甚至写了 "Mandatory recall step"——每次涉及历史信息时，**必须**调用记忆搜索。

但实际运行数据告诉了我另一个故事：

- 242 个会话中，只有 30 个调用过记忆搜索工具——**12.4%**
- 其他 Agent（开发经理、工程师等）的调用率是 **0%**

87.6% 的时间里，这套精心搭建的记忆检索系统在空转。

这让我开始思考一个被大家忽略的问题：**AI Agent 真的需要"记住"那么多东西吗？**

## 两条路线：记得多 vs 记得少

当前 AI Agent 的记忆系统大致分为两个流派。

**"记得多"派**以 OpenClaw 为代表。它的记忆架构是一个多层系统：

- 工作区文件全量注入（MEMORY.md、SOUL.md、USER.md 等，约 30-40KB）
- 每日日志自动加载（today + yesterday）
- 向量搜索按需召回（memory_search）
- 上下文压缩前自动保存（Memory Flush）
- 历史会话索引（Session Transcript）

理念是：**信息越多，Agent 越聪明。** 用检索引擎保证海量记忆中能找到需要的。

**"记得少"派**以 Claude Code 为代表。它的记忆系统极其简单：

- CLAUDE.md：用户写的项目指令，建议不超过 200 行
- Auto Memory：AI 自己写的笔记，每次加载前 200 行
- 没有向量搜索，没有 embedding，没有索引

理念是：**200 行够了，维护好这 200 行比建搜索引擎重要。**

一个建设了完整的记忆基础设施，一个几乎没有。但从实际效果看，两者的用户体验差距并没有想象中那么大。

这是为什么？

## 模型不搜索，可能是对的

回到 12.4% 这个数字。为什么模型不愿意调用搜索工具？

最直接的原因是：**它已经"看到"了大部分需要的信息。**

OpenClaw 启动时会把 MEMORY.md（477 行、19KB）全量注入上下文。加上其他工作区文件，模型在开始对话前就已经拥有了 30-40KB 的背景知识。对于 200K token 的上下文窗口来说，这大约占 15-20%。

既然信息已经在眼前了，为什么还要费力去搜索？这不是偷懒，是理性选择。

但更深层的原因可能是：**模型在做一种隐式的"该不该搜"的判断。**

学术界有一篇相关研究——Zero-RAG（2025）发现，当模型的参数知识已经包含某个事实时，再通过检索注入相同信息不仅无益，反而可能因为表述差异产生冲突，降低准确率。他们设计了一个 Query Router：先判断模型是否"已知"，已知就跳过检索。去掉这个 Router 后性能明显下降。

也就是说，**"不该搜的时候硬搜"是有害的。**

## 无关记忆的真实影响

这就引出了一个更根本的问题：如果我们强制注入或召回记忆，无关的记忆会影响推理质量吗？

答案比想象中复杂。

Cuconasu 等人在 2024 年 SIGIR 的一篇论文中发现了一个反直觉的现象：**在 RAG 系统中加入完全无关的随机文档，准确率反而提升了 30% 以上。**

但后续研究（Amiraz et al., 2025）将"无关信息"做了更精细的分类，揭示了关键区别：

- **Random（随机无关）**：和查询完全不搭边的内容——几乎无害，有时甚至有益
- **Distracting（干扰性无关）**：和查询语义相关但不包含正确答案——**显著降低准确率**

区别在哪？一条关于"昨天晚餐吃了什么"的记忆，在你讨论系统架构时完全无害——模型知道这不相关，会自动忽略。但一条三个月前关于"这个配置应该这样写"的旧笔记，如果被语义搜索召回，就变成了高浓度的 Distracting 信息——因为它**看起来像是答案，但其实已经过时了。**

巧合的是，**记忆搜索系统天然倾向于产生第二类信息**。语义搜索的本质就是找"相关"的内容，而这些"相关但不完全对"的结果恰恰是最危险的。

再叠加经典的 "Lost in the Middle" 效应（Liu et al., 2023）——模型对上下文的利用呈 U 型曲线，开头和结尾的信息利用率最高，中间容易被忽略——大量注入的记忆在上下文膨胀后会逐渐"沉"到注意力盲区，既浪费了 token，又没有被有效利用。

## 一个不太一样的视角

讨论到这里，一个不太被提及的角度浮现了出来。

我们一直在讨论"怎么让 AI 记住更多"——更好的 embedding、更精确的检索、更大的索引。但也许问题的根源不在这里。

**AI Agent 的对话式交互，每天产生的真正需要记忆的新增信息，其实极少。**

想想看，一个 AI Agent 一天的工作大概是：帮你查个天气、改个配置、讨论一下技术方案、提醒你开会。这些对话中，真正需要跨会话记住的新信息可能就几条：

- 某个配置从 A 改成了 B
- 你做了一个决定：用方案 X 不用方案 Y
- 你提到了一个新的偏好

而通用知识——Python 怎么写、K8s 怎么部署、什么天气适合穿什么衣服——模型参数里本来就有，不需要"记忆"。

所以 Agent 的记忆需求，可能只是一个**很薄的增量层**：你的上下文、你的偏好、你的决定。框架性地知道"有这件事存在"就够了，细节可以在需要时重新获取或推导。

这很像人类的记忆方式。

## 人类怎么记忆

人一天接收的信息量是巨大的——视觉、听觉、对话、阅读、情绪、身体感受。但晚上睡觉时，大脑做的事情不是"把今天的信息全部存档"，而是：

1. **重组**：把碎片信息整合成结构化的理解
2. **压缩**：只保留框架和关键要点
3. **遗忘**：丢弃绝大部分细节

第二天醒来，你记得的可能就是"昨天和老王讨论了架构方案，他倾向方案 B"。讨论的具体措辞、当时的表情、午餐吃了什么——全忘了。但这不妨碍你在今天继续推进工作。

如果需要细节呢？你去翻会议纪要、翻聊天记录、再问一遍。外部存储补充内部记忆。

现在的 AI Agent 记忆系统在试图做什么？**它们在试图让 Agent "永不遗忘"。** 每条日志保留、每段对话索引、每个细节可搜索。这恰恰和人类记忆的运作方式相反。

也许 Agent 也需要"睡眠"——一个后台运行的记忆整理过程：

- 回顾一天的对话，提取框架性要点
- 把过时的信息标记为"可遗忘"
- 合并重复的记忆条目
- 保留的结果可能就十几行

OpenClaw 的 PCEC（自我进化引擎）和心跳巡检机制其实已经触碰到了这个方向——定期审视系统状态、整理记忆文件、更新长期记忆。但目前的重心仍然在"记住更多"，而不是"忘掉该忘的"。

## 没有答案，只有方向

坦率地说，AI Agent 的记忆系统目前仍处于探索阶段。没有最优解，可能暂时也没有更优解——只有在使用过程中逐步积累的经验和猜测。

但有几个方向值得持续思考：

**Less is more 可能不只是口号。** 对于理解大模型的开发者来说，记忆系统的复杂度可以接受。但对于绝大多数用户来说，能"无感知地用好记忆"才是真正重要的。200 行自动维护的笔记可能比一套完整的 RAG 管线更有用。

**搜索精度比搜索频率重要。** 与其让 Agent 更频繁地搜索记忆，不如确保搜出来的每条都是高质量的。一条过时的"相关"记忆造成的危害，可能大于十条缺失的记忆。

**遗忘可能和记忆一样重要。** 人类大脑不是存储容量不够才遗忘，而是遗忘本身就是认知功能的一部分——它帮助我们从噪音中提取信号，保持思维的清晰度。也许 AI Agent 的记忆系统也需要一个"遗忘引擎"，和"检索引擎"同等重要。

**Context window 的持续扩大会改变一切。** 当上下文窗口从 200K 扩展到 1M、10M，"全量注入"可能成为最简单也最有效的方案——根本不需要搜索，全塞进去就行。那时候记忆系统的核心问题就不再是"怎么找到需要的信息"，而是"怎么在海量上下文中不被噪音淹没"。本质上还是一个遗忘问题。

也许有一天，AI Agent 记忆系统的终极形态会变得非常简单：

**一个薄薄的笔记本，加上一个知道该划掉哪些内容的橡皮擦。**
